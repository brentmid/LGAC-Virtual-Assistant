LGAC Virtual Assistant

Transition Brief for Club IT

A RAG-powered Q&A assistant for club members — what it is, what it costs, and how to own it.

Use the arrow keys → or click to advance · press Esc for an overview

What it is

A question-and-answer chatbot for club members.

  • Members ask in plain language; it answers only from official club documents (13 PDFs/DOCX: dress code, dues, rules, hours, membership…).
  • It does not browse the web and is instructed to decline off-topic questions and never fabricate answers.
  • Every answer cites the source document it came from.

Built as an MVP proof-of-concept to put in front of testers, then iterate.

How it works

1 · Offline indexing

Club documents are read, cleaned, split into chunks, and converted to a searchable index once, at build time. No live database to run.

2 · Answering a question

The question is matched against the index; the most relevant passages plus the question are sent to Claude, which writes the answer with citations.

Python · FastAPI ChromaDB (embedded) Claude Sonnet 4.6 Docker · Google Cloud Run

The model is a single configuration setting. Over time the club can switch to a newer Sonnet release — or an Opus model — for higher answer quality, traded off against higher per-question cost.

Where it lived (test deployment)

  • Ran on Google Cloud Run — automatic HTTPS, scales to zero (≈ $0 when idle).
  • Sizing: 1 CPU / 1 GB RAM, simple shared-password gate.
  • Cost while running: < $5/month hosting + ~$0.01–0.03 per question (Claude API).
⚠️ The test deployment ran on brentmid's personal accounts and has been shut down to stop billing. All of its credentials are dead.

The starting point for Club IT

Nothing is running right now — and that's intentional.

  • The personal GCP project, API key, and passwords are decommissioned.
  • To stand it back up, Club IT provisions their own: a Google Cloud project, an Anthropic API key, and new passwords.
  • The full step-by-step runbook ships with the code (DEPLOYMENT_PLAN.md) and is project-portable.

Honest gaps — it's an MVP

Deliberate shortcuts taken to ship fast. Each must be addressed before opening to the full membership.

GapWhy it matters in production
Single shared passwordNo per-member identity or revocation
Sessions held in memoryConversations lost on restart; breaks across instances
No rate limitingOne user could run up unbounded API cost
No monitoring / loggingNo visibility into usage, errors, or spend
Doc updates need a rebuildNon-technical staff can't update content themselves

Plus: no OCR for scanned docs, no streaming responses. All tracked as GitHub issues.

brentmid can help address these if desired — see the next slide for ownership options.

The key decision: who owns it?

A · Club owns all

Club's own cloud + API accounts; brentmid steps away.

Full control · needs technical staff/vendor.

B · brentmid runs, club funds

brentmid keeps operating it; club pays for API + hosting.

Fastest live · depends on one person.

C · Managed vendor

A third party operates it under a support contract.

Defined accountability · recurring cost.

Account transfer checklist

  • Create a club-owned Google Cloud project and Anthropic API key (with a spend cap).
  • Set fresh passwords in the new project's secret store — reuse nothing.
  • Redeploy using the included runbook against the club's project.
  • Point a club subdomain at the new service; set a budget alert.
  • Transfer the GitHub repository to a club-owned account (if desired).

Roadmap to production

Phase 0 — Account transfer & budget alerts ~1–2 days · prerequisite

Phase 1 — Real member authentication ~3–8 days · replaces the shared password

Phase 2 — Persistent sessions & feedback ~2–4 days · survive restarts, scale out

Phase 3 — Rate limiting & abuse controls ~1–3 days · bound API spend

Phase 4 — Monitoring, logging & cost visibility ~2–4 days

Phase 5 — Self-service document updates ~4–8 days · no rebuild needed

Phase 6 — Quality & UX polish ongoing · streaming, OCR, tests

Phases 0–4 (~9–21 developer-days) get to a defensible production system; 5–6 follow launch.

Authentication: the open question

The club's system of record is Northstar (globalnorthstar.com).

First to determine: does Northstar expose an API for authentication? (Likely not.)

If not, an approach to investigate:

Passcode

Issue and manage member access codes for the assistant.

OAuth login

Create and distribute an OAuth-based member sign-in.

Site-to-site

Members authenticate to Northstar at thelandings.com first, then reach the assistant on a short-lived token.

Options still to be investigated — brentmid can help evaluate.

≈ $10–30/month in API cost at 1,000 questions/month, plus < $5/month hosting.

Set an Anthropic monthly spend cap and a cloud budget alert — both take minutes.

Operating it, day to day

  • Custom domain: map a club subdomain (e.g. assistant.thelandings.com); TLS is automatic.
  • Updating documents: today, replace files and redeploy; Phase 5 adds a self-service upload page.
  • Backups: the only irreplaceable data is the source documents — everything else rebuilds from them.
  • Cost & quality: watch the billing report and spend cap; log questions (with a privacy review) to gauge answer quality.

Recommended sequence

  1. Decide ownership (A / B / C) — everything else follows this.
  2. Phase 0: transfer accounts, set spend caps — now club-owned.
  3. Phases 1–3: auth, persistent sessions, rate limiting — the gate to a full launch.
  4. Soft launch to a member subset; turn on monitoring; watch cost & quality.
  5. Phases 5–6: self-service docs and polish once proven.

Open questions for the club

  • Who owns day-to-day operations — internal IT, brentmid, or a vendor?
  • What monthly budget ceiling for API + hosting?
  • Is logging member questions for quality review acceptable under club privacy norms?
  • What subdomain should the service live at?
